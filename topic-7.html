Topic 7: Proposals to Other Teams

Propuestas a Otros Equipos • Medium

Why This Matters / Por que importa

They told you: "it also involves making proposals to other teams." You need to define HOW other teams' services integrate with the orchestrator. This is about contracts and interfaces.

Te dijeron que tambien implica propuestas a otros equipos. Necesitas definir COMO se integran. Esto es sobre contratos e interfaces.

The Standard Step Interface

Every external service should follow the SAME interface. This makes the orchestrator simple — no special code per service.

Cada servicio debe seguir la MISMA interfaz. Asi el orquestador es simple — sin codigo especial por servicio.

REQUEST (what the orchestrator sends):

POST /execute
{
  "workflow_id": "abc-123",
  "step_id": "ocr",
  "attempt": 1,
  "inputs": {
    "document_image": "s3://bucket/doc.jpg",
    "country": "DE",
    "document_type": "passport"
  }
}


RESPONSE (what the service returns):

{
  "status": "completed",
  "outputs": {
    "name": "Jane Doe",
    "date_of_birth": "1990-01-01",
    "confidence": 0.97,
    "face_crop_uri": "s3://bucket/face.jpg"
  },
  "error": null
}

What to Propose to Each Team

To the OCR Team

  • Accept a document image URI and return extracted fields
  • Include a confidence score (0-1) for each field
  • Extract and store the face crop separately (needed for face matching)
  • Return structured errors: retryable vs permanent
  • Be idempotent — same image in = same result out

To the Biometrics Team (Liveness + Face Match)

  • Two separate endpoints: Liveness and Face Match
  • Liveness: accepts selfie/video, returns alive=bool + confidence + best frame URI
  • Face Match: accepts two face images, returns match_score (0-1)
Why separate?

The orchestrator runs liveness in parallel with OCR. Face match runs AFTER both. If they're one service, you lose that parallelism.

El orquestador ejecuta liveness en paralelo con OCR. Face match corre DESPUES. Si son un servicio, pierdes el paralelismo.

To the Compliance Team (Sanctions)

  • Accept name + date of birth + nationality
  • Return hit=bool + match details
  • Support async for checks >5s (return 202 + callback)
  • Include a 'requires_manual_review' flag for fuzzy matches

Error Contract / Contrato de errores

Every service should return errors in the same format:

{
  "status": "failed",
  "error": {
    "code": "DOCUMENT_UNREADABLE",
    "message": "Could not extract text from document image",
    "retryable": false,
    "category": "permanent"
  }
}

Error categories:
  "transient"  → orchestrator should retry
  "permanent"  → don't retry, step has failed
  "degraded"   → service partially worked, results incomplete

Without a standard error contract, the orchestrator needs custom error handling per service. With it, the retry logic is generic.

Sin un contrato estandar, necesitas manejo personalizado por servicio. Con el, la logica de reintento es generica.

Monorepo Structure / Estructura del monorepo

They mentioned a monorepo. Propose a package structure:

packages/
  workflow-engine/          ← core DAG executor (what you built)
  workflow-api/             ← REST API for customers
  workflow-persistence/     ← database layer
  workflow-dashboard/       ← ops UI for monitoring
  step-contracts/           ← shared types for step interface
                              (OTHER teams import this package)
  decision-engine/          ← business logic for decisions
Key insight

The step-contracts package is shared with other teams. When you change the contract, everyone updates together because it's a monorepo — no version mismatch issues.

El paquete step-contracts se comparte. Cuando cambias el contrato, todos actualizan juntos porque es un monorepo.

"How would you ensure other teams' services integrate cleanly?"
"I'd define a standard step interface — a shared contract that every service must follow. Request: workflow ID, step ID, inputs. Response: status, outputs, structured error. The contract lives in a shared package in the monorepo. This means the orchestrator doesn't need service-specific code. The retry logic, timeout handling, and data passing are all generic. If a new service doesn't follow the contract, it's a conversation, not a code change in the orchestrator."
"Definiria una interfaz estandar — un contrato compartido en el monorepo. El orquestador no necesita codigo especifico por servicio. Si un nuevo servicio no sigue el contrato, es una conversacion, no un cambio de codigo."

SDK proposal / Propuesta para el SDK

The SDK should handle: image capture, compression, upload to object storage, and then call the orchestrator API with image URIs. The SDK should NOT send raw image bytes in the API call — upload first, then pass references.

El SDK debe manejar: captura de imagen, compresion, subida a object storage, y luego llamar a la API del orquestador con URIs. El SDK NO debe enviar bytes de imagen en la llamada API — subir primero, luego pasar referencias.

Compliance requirements for the API

  • All data stays in EU — object storage and database must be in EU regions (eIDAS 2.0, AMLR)
    Todos los datos se quedan en la UE — almacenamiento en regiones de la UE
  • Every API call is logged — who called what, when, with what result (audit trail)
    Cada llamada a la API se registra — quien llamo que, cuando, con que resultado
  • Data retention policies — verification data must be kept for X years, then deleted (GDPR)
    Politicas de retencion — datos de verificacion se guardan X anos, luego se borran
  • Authentication — every customer has API keys; every internal service uses service-to-service auth
    Autenticacion — cada cliente tiene API keys; cada servicio interno usa auth servicio-a-servicio
