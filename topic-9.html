Topic 9: HTTP & Communication Patterns

HTTP y Patrones de Comunicacion • Medium

What is HTTP? / Que es HTTP?

HTTP is the language that computers use to talk to each other over the internet. When your browser opens a page, it uses HTTP. When your orchestrator calls the OCR service, it uses HTTP too.

HTTP es el idioma que usan las computadoras para hablar entre si por internet.

HTTP Methods — The Verbs / Los verbos

Think of them as actions — what you want to do:

MethodWhat it doesAnalogyIn our system
GETRead/fetch data"Show me the menu"GET /workflows/abc-123
POSTCreate/send something new"I want to order this"POST /verifications
PUTReplace entirely"Replace my whole order"Update a workflow definition
PATCHUpdate partially"Change just the drink"Update one field
DELETERemove something"Cancel my order"Cancel a verification
Rule of thumb / Regla general

Reading something? → GET. Sending data to create or process? → POST. All step calls are POST because we SEND data to be processed.

Leyendo algo? GET. Enviando datos para crear o procesar? POST. Todas las llamadas a pasos son POST porque ENVIAMOS datos para procesar.

Status Codes / Codigos de estado

2xx = SUCCESS (todo bien)
  200 OK              → here's your data
  201 Created         → I created the thing you asked
  202 Accepted        → got it, working on it (async)

4xx = CLIENT ERROR (tu culpa)
  400 Bad Request     → you sent bad data
  401 Unauthorized    → who are you?
  404 Not Found       → doesn't exist
  429 Too Many Requests → slow down! (rate limit)

5xx = SERVER ERROR (culpa del servidor)
  500 Internal Error  → something broke
  502 Bad Gateway     → service behind me is down
  503 Unavailable     → I'm overloaded, try later
For the orchestrator

4xx → permanent failure, don't retry (bad input). 429, 5xx → transient failure, retry with backoff.

4xx → fallo permanente, no reintentar. 429, 5xx → fallo transitorio, reintentar con backoff.

What is a Timeout? / Que es un Timeout?

A timeout is a maximum wait time. "I'll wait 10 seconds for your answer. If you don't respond, I give up."

Un timeout es un tiempo maximo de espera. "Espero 10 segundos tu respuesta. Si no respondes, me rindo."

WITHOUT timeout (dangerous):
  Orchestrator → calls OCR service
  Orchestrator waits...
  Orchestrator waits...
  Orchestrator waits... forever??
  (If the service is dead, you wait FOREVER)

WITH timeout (safe):
  Orchestrator → calls OCR service
  Orchestrator waits...
  10 seconds pass... no response
  Orchestrator: "Time's up. I'll retry or fail."
  (You always get an answer within a known time)

Analogy: You order food at a restaurant. Without timeout: you sit forever. With timeout: "If my food isn't here in 30 min, I'm leaving."

Analogia: Pides comida. Sin timeout: esperas para siempre. Con timeout: "Si no llega en 30 min, me voy."

Querying — Getting Only What You Need

Query Parameters (simplest way to filter)

GET /verifications?status=pending
  → only return pending verifications

GET /verifications?status=completed&country=DE
  → completed verifications from Germany

GET /verifications/abc-123?fields=status,verdict
  → only return status and verdict, not everything

REST vs GraphQL

REST (server decides what to return):
  GET /verifications/abc-123
  Response: { id, status, verdict, created_at, updated_at,
              steps: [...everything...], customer: {...},
              signer: {...}, documents: [...] }
  → returns EVERYTHING even if you only need status

GraphQL (YOU decide what to return):
  POST /graphql
  { query: "{ verification(id: \"abc-123\") { status verdict } }" }
  Response: { status: "COMPLETED", verdict: "APPROVED" }
  → returns ONLY what you asked for
For this system: REST is enough

The API surface is small — a few endpoints with predictable shapes. GraphQL adds complexity (schema management, resolvers, caching challenges) without solving a real problem. REST with query parameters for filtering covers the use case.

La superficie de API es pequena — pocos endpoints con formas predecibles. GraphQL agrega complejidad sin resolver un problema real. REST con parametros de consulta cubre el caso de uso.

Real-Time Communication Patterns

Different ways to get live updates:

1. Polling (simplest)

Client asks "are you done yet?" over and over.

El cliente pregunta "ya terminaste?" una y otra vez.

Client                          Server
  |── GET /status ──────────────→|
  |←── { status: "running" } ───|
  |   (wait 2 seconds)           |
  |── GET /status ──────────────→|
  |←── { status: "running" } ───|
  |   (wait 2 seconds)           |
  |── GET /status ──────────────→|
  |←── { status: "completed" } ─|   done!

Analogy: Calling the restaurant every 2 min
         "Is my food ready?" "No." "Ready?" "Yes!"
ProsCons
Dead simple, works everywhereWastes network calls, delay between updates

2. Long Polling (smarter)

Client asks, server holds the connection open until there's news.

El cliente pregunta, el servidor mantiene la conexion abierta hasta que hay noticias.

Client                          Server
  |── GET /status ──────────────→|
  |         (server holds...)    |
  |         (something happens!) |
  |←── { status: "completed" } ─|   immediate!
  |── GET /status ──────────────→|   (new request for next update)

Analogy: Call restaurant, they say
         "Stay on the line, I'll tell you when it's ready"
ProsCons
Fewer requests, near real-timeServer holds connections, timeout needed

3. Server-Sent Events / SSE (one-way stream)

Server pushes updates to the client. Client just listens.

El servidor empuja actualizaciones. El cliente solo escucha.

Client                          Server
  |── GET /events (keep open) ──→|
  |←── data: step "ocr" started  |
  |←── data: step "ocr" completed|
  |←── data: step "face_match"...|
  |←── data: verdict "APPROVED"  |
  Connection stays open, server keeps pushing

Analogy: A radio — you tune in and listen
ProsCons
Real-time, built into browsers, simple HTTPOne direction only (server → client)

4. WebSockets (two-way real-time)

A persistent connection where both sides can send anytime.

Una conexion persistente donde ambos lados pueden enviar en cualquier momento.

Client                          Server
  |── HTTP upgrade to WebSocket ─→|
  |←── Connection established ───|
  |←── { step: "ocr", "started" }|
  |── { action: "cancel" } ─────→|    client can send too!
  |←── { workflow: "cancelled" } |

Analogy: A phone call — both people can talk anytime
ProsCons
True real-time, both directions, low latencyMost complex, needs special server, harder to scale

Which Pattern for Our System?

CommunicationBest PatternWhy
Customer backend ←→ Platform Webhook/callback Platform calls customer's URL when done. Simple, reliable.
End user app ←→ Platform SSE User watches progress: "Checking..." → "Approved!" One direction enough.
Video-assisted session WebSockets User and agent interact in real-time. Two-way needed.
Orchestrator → Services REST request/response Orchestrator calls, waits, gets result. No real-time needed.
Summary / Resumen

Everything in this system is REST except: SSE for user-facing live progress, and WebSockets only if there's a real-time video session. Don't over-engineer.

Todo en este sistema es REST excepto: SSE para progreso en vivo al usuario, y WebSockets solo si hay sesion de video en tiempo real. No sobre-ingenieries.

"How would the end user see the progress of their verification?"
"SSE — Server-Sent Events. The mobile SDK opens an SSE connection when the user starts verification. The orchestrator pushes events as each step completes: 'Analyzing document...', 'Checking identity...', 'Approved!'. It's one-directional, built into browsers, and simpler than WebSockets. We'd only need WebSockets for a video-assisted flow where the user and agent interact in real-time."
"SSE — Server-Sent Events. El SDK abre una conexion SSE cuando el usuario inicia la verificacion. El orquestador empuja eventos cuando cada paso completa. Es unidireccional, integrado en browsers, y mas simple que WebSockets. Solo necesitariamos WebSockets para un flujo de video donde usuario y agente interactuan en tiempo real."
