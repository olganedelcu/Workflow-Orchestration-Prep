Topic 6: Conditional Branching

Ramas Condicionales • Medium

The Problem / El problema

Your current engine has a FIXED DAG — every step always runs (unless a dependency fails). In reality, verification flows need to take DIFFERENT PATHS based on results.

Tu motor actual tiene un DAG FIJO. En realidad, los flujos necesitan tomar CAMINOS DIFERENTES basados en resultados.

Remember the InstantSign diagram:

Check ID validity
       |
  ┌────┴─────┐
  |          |
ID VALID   ID NOT VALID
  |          |
  v          v
 Sign     Full verification
           (VideoIdent/eID)
            then Sign

The path depends on the RESULT of the ID check. Your engine can't do this yet.

The Solution: Guard Conditions on Edges

BEFORE (your current engine):
  step: { id: "check_id", next: ["sign", "full_verification"] }
  // BOTH always execute

AFTER (with conditions):
  step: {
    id: "check_id",
    next: [
      { stepId: "sign",              when: "output.id_valid == true"  },
      { stepId: "full_verification", when: "output.id_valid == false" }
    ]
  }
  // Only ONE executes based on the condition

How It Works in the Orchestrator

1. Step "check_id" completes with output: { id_valid: true }

2. Orchestrator looks at its "next" edges

3. Evaluates each condition:
     "output.id_valid == true"  → TRUE  → "sign" is ACTIVATED
     "output.id_valid == false" → FALSE → "full_verification" is SKIPPED

4. Only activated steps become eligible to run

5. Skipped steps are marked as "SKIPPED" (new status!)

The Diamond Pattern with Conditions

They mentioned diamonds!

The interviewer specifically mentioned a diamond workflow pattern.

SIMPLE DIAMOND (no conditions, what your engine already does):

        A
       / \
      B   C     ← both always run in parallel
       \ /
        D       ← waits for BOTH B and C

CONDITIONAL DIAMOND:

        A
       / \
      B   C     ← only one runs, based on A's output
       \ /
        D       ← waits for whichever branch was ACTIVATED

The challenge: D depends on B and C, but only ONE ran. How does D know it's ready?

El reto: D depende de B y C, pero solo UNO corrio. Como sabe D que esta listo?

Solution
BEFORE: step ready if ALL dependencies are COMPLETED
AFTER:  step ready if all dependencies are COMPLETED or SKIPPED

New Step Status: SKIPPED

Current engine: WAITING → RUNNING → COMPLETED
                                  → FAILED

With branching: WAITING → RUNNING → COMPLETED
                                  → FAILED
                WAITING → SKIPPED  (condition was false)

Real-World Example

Different countries require different verification methods:

workflow: identity_verification
steps:
  - id: document_capture
    next:
      - stepId: ocr
        when: "always"

  - id: ocr
    next:
      - stepId: video_ident
        when: "country in ['DE','AT'] AND ocr.confidence < 0.8"
      - stepId: auto_ident
        when: "country in ['DE','AT'] AND ocr.confidence >= 0.8"
      - stepId: eid_check
        when: "country in ['FR','ES']"

  - id: video_ident     # agent-assisted video call
    next: [decision]

  - id: auto_ident      # fully automated
    next: [decision]

  - id: eid_check       # electronic ID
    next: [decision]

  - id: decision
    # waits for whichever verification method was activated
"How would you handle different verification flows for different countries?"
"Two levels. First, different workflow definitions per product or region — Germany might use a different DAG than France. Second, conditional branching within a workflow — based on OCR confidence, the flow might take the automated path or route to video verification. The orchestrator supports both: it selects the right workflow definition at creation time, and evaluates conditions at runtime as steps complete."
"Dos niveles. Primero, diferentes definiciones de workflow por region. Segundo, ramas condicionales dentro del workflow. El orquestador soporta ambos: selecciona la definicion correcta al crear, y evalua condiciones en tiempo de ejecucion."
