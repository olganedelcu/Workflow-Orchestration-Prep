Topic 2: Calling External Services

Llamando Servicios Externos • High Priority

The Problem / El problema

Your current engine calls simulateStepExecution() — a fake function. In reality, each step calls a real external service over the network. The network is unreliable. Services are slow. Things break.

Tu motor actual llama a una funcion falsa. En realidad, cada paso llama a un servicio externo real por la red. La red no es confiable. Los servicios son lentos. Las cosas se rompen.

Three Patterns / Tres patrones

Pattern 1: Synchronous HTTP (Simple)

Orchestrator                    OCR Service
     |                              |
     |---- POST /analyze ---------->|
     |         (waits...)           |
     |         (waits...)           |
     |<--- 200 OK + results --------|
     |                              |
     |  Now orchestrator continues

The orchestrator sends a request and WAITS for the response. Like calling someone on the phone and staying on the line.

El orquestador envia una peticion y ESPERA la respuesta. Como llamar a alguien por telefono y quedarse en la linea.

ProsCons
Simple to implementOrchestrator is blocked waiting
Easy to debugIf service takes 30s, you wait 30s
No extra infrastructureCan't handle many concurrent workflows

Pattern 2: Async with Callback (Medium)

Orchestrator                    OCR Service
     |                              |
     |---- POST /analyze ---------->|
     |<--- 202 Accepted ------------|  (immediate "got it")
     |                              |
     |  (orchestrator is FREE       |
     |   to do other work)          |
     |                              |  (service processes...)
     |                              |
     |<--- POST /callback ---------|  (service calls back when done)
     |     {results}                |

Like sending a text message — you don't wait for a reply.

Como enviar un mensaje de texto — no esperas la respuesta.

ProsCons
Orchestrator not blockedWhat if callback never arrives?
Handles slow services wellMore complex
ScalableNeed to match callback to workflow

Pattern 3: Message Queue (Most decoupled)

Orchestrator         Queue              OCR Worker
     |                 |                    |
     |-- publish ---->|                    |
     |  {analyze this} |                    |
     |                 |--- deliver ------->|
     |                 |                    | (processes...)
     |                 |<-- publish result -|
     |<-- consume -----|                    |
     |  {results}      |                    |

Like leaving a letter in a mailbox.

Como dejar una carta en un buzon.

ProsCons
Fully decoupledMore infrastructure (need a queue)
Built-in retryHarder to debug
Handles backpressureAdded latency
If worker crashes, message stays in queue

What to Propose / Que proponer

Your answer

"I'd start with synchronous HTTP with timeouts. It's the simplest approach and good enough for a greenfield project. Each step call: send request, wait up to N seconds, handle response or timeout. If a specific service proves slow or unreliable, we migrate that ONE integration to async without changing the whole system."

"Empezaria con HTTP sincrono con timeouts. Es el enfoque mas simple y suficiente para un proyecto greenfield. Si un servicio resulta lento, migramos ESA integracion a asincrono sin cambiar todo el sistema."

Why this is a good answer: It shows you don't over-engineer. You pick the simplest thing that works and evolve when needed.

Muestra que no sobre-ingenierias. Eliges lo mas simple que funciona y evolucionas cuando es necesario.

Timeout Handling / Manejo de timeouts

Request sent ──────── 10s timeout ──────── ?
                          |
                  ┌───────┴────────┐
                  |                |
            Response arrives    No response
            before timeout      TIMEOUT
                  |                |
                  v                v
             Continue          Retry? Fail?

Use Promise.race:

const result = await Promise.race([
  httpCall(serviceUrl, inputs),     // the actual call
  timeout(10000)                    // 10 second bomb
]);
// Whichever finishes first wins
"What if the OCR service takes 10 seconds sometimes and 100ms other times?"
"I'd use sync HTTP with a generous timeout — say 15 seconds for OCR. The orchestrator already runs independent steps in parallel, so while OCR is slow, Liveness runs at the same time. If OCR consistently exceeds the timeout, I'd add retry logic for that specific step. If the problem persists, that's when I'd consider moving OCR to async with a callback, but only for that one service."
"Usaria HTTP sincrono con un timeout generoso — digamos 15 segundos para OCR. El orquestador ya ejecuta pasos independientes en paralelo. Si OCR consistentemente excede el timeout, agregaria logica de reintento. Si persiste, ahi consideraria mover OCR a asincrono, pero solo ese servicio."
