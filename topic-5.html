Topic 5: Persistence & Recovery

Persistencia y Recuperacion • Medium

The Problem / El problema

Your current engine keeps everything in memory. If the process crashes, all running workflows are lost.

Tu motor actual mantiene todo en memoria. Si el proceso se cae, todos los workflows se pierden.

Workflow running in memory:
  Step 1: COMPLETED
  Step 2: COMPLETED
  Step 3: RUNNING...
                      CRASH!

After restart:
  ??? Everything is gone

The Solution: Persist After Each Step

Workflow running:
  Step 1: COMPLETED  → save to DB
  Step 2: COMPLETED  → save to DB
  Step 3: RUNNING...
                      CRASH!

After restart:
  Load from DB:
    Step 1: COMPLETED (with outputs)
    Step 2: COMPLETED (with outputs)
    Step 3: WAITING   (re-run this one)

  Resume from Step 3!

Database Schema / Esquema de base de datos

-- The workflow instance (one per verification)
workflow_instances:
  id              UUID        -- "abc-123"
  definition_id   TEXT        -- "identity_verification_v2"
  status          TEXT        -- "RUNNING", "COMPLETED", "FAILED"
  context_data    JSONB       -- the WorkflowContext (inputs, step outputs)
  created_at      TIMESTAMP
  updated_at      TIMESTAMP

-- Each step execution (one per step per workflow)
step_executions:
  id              UUID
  workflow_id     UUID        -- FK to workflow_instances
  step_id         TEXT        -- "ocr", "face_match", etc.
  status          TEXT        -- "WAITING", "RUNNING", "COMPLETED", "FAILED"
  inputs          JSONB       -- what was sent to the service
  outputs         JSONB       -- what the service returned
  error           TEXT        -- error message if failed
  attempt_number  INT         -- which retry attempt (1, 2, 3...)
  start_clock     INT         -- from your logical clock
  finished_clock  INT
  started_at      TIMESTAMP
  finished_at     TIMESTAMP

Why PostgreSQL / Por que PostgreSQL

ReasonExplanation
ACID transactionsEach step completion is atomic — never a half-written state
JSONB columnsFlexible step inputs/outputs without rigid schemas
Mature & reliableBattle-tested, huge community
Good enough for scaleIdentity verification is not millions/second
Greenfield projectNo existing DB to integrate with
Tip

Don't propose exotic databases unless they ask. PostgreSQL is the safe, correct answer for this use case.

Event Sourcing

Instead of updating rows, append events. Each state change is a new row.

En vez de actualizar filas, agregar eventos. Cada cambio de estado es una nueva fila.

workflow_events:
  id, workflow_id, event_type, step_id, payload, clock, created_at

Example events:
  { event: "WORKFLOW_CREATED", workflow_id: "abc" }
  { event: "STEP_STARTED",    step: "ocr",        clock: 0 }
  { event: "STEP_STARTED",    step: "liveness",   clock: 0 }
  { event: "STEP_COMPLETED",  step: "ocr",        clock: 1, outputs: {...} }
  { event: "STEP_COMPLETED",  step: "liveness",   clock: 1, outputs: {...} }
  { event: "STEP_STARTED",    step: "face_match", clock: 2 }
  ...

Why this matters for the platform

  1. Audit trail for free — regulators can see exactly what happened, when, and why. You never delete or modify history.

    Pista de auditoria gratis — los reguladores pueden ver exactamente que paso.

  2. Debugging — replay events to understand what happened in a failed verification.

    Depuracion — reproduce eventos para entender que paso.

  3. Recovery — rebuild workflow state by replaying events.

    Recuperacion — reconstruir estado reproduciendo eventos.

What to propose

"For v1, simple CRUD — update step_executions after each step. But also insert into an events table for audit. This gives us the audit trail immediately without the full complexity of event sourcing."

"Para v1, CRUD simple. Pero tambien insertar en una tabla de eventos para auditoria. Esto nos da la pista de auditoria sin la complejidad completa de event sourcing."

"If the system crashes mid-verification, what happens?"
"After each step completes, we persist the result and outputs to the database in a transaction. On crash recovery, we load incomplete workflows, check which steps have completed, and resume from where we left off. Since steps are idempotent, re-running a step that might have partially executed is safe. The logical clock values help us maintain ordering guarantees across the recovery."
"Despues de cada paso, persistimos resultado y outputs en la BD. Al recuperarnos, cargamos workflows incompletos, verificamos que pasos completaron, y retomamos. Como los pasos son idempotentes, re-ejecutar es seguro. El reloj logico mantiene garantias de ordenamiento."
