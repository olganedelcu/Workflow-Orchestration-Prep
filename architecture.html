Layer 1: Client Layer / Capa del Cliente

These are the consumers of your platform. They don't talk to services directly — they only talk to YOUR API.

Estos son los consumidores de tu plataforma. No hablan con los servicios directamente — solo hablan con TU API.

Client / ClienteWhat it does / Que haceHow it connects / Como se conecta
Mobile SDK (iOS/Android) Captures document photos + selfie on the user's device. Compresses and uploads images to object storage. REST: POST /verifications with image URIs
Web SDK Same as mobile but runs in the browser. Uses the device camera via WebRTC. REST: POST /verifications with image URIs
Customer Backend The customer's server (e.g. a bank). Receives results via webhook when verification completes. REST: GET /verifications/:id + webhook callback
Key design decision / Decision clave

The SDK uploads images to object storage FIRST, then sends URIs to the API. This means the API never handles raw image bytes — only references. Faster, lighter, scalable.

El SDK sube imagenes al object storage PRIMERO, luego envia URIs a la API. La API nunca maneja bytes de imagen — solo referencias. Mas rapido, ligero, escalable.

SDK Flow / Flujo del SDK:

  1. User opens app → SDK initializes
     Usuario abre app → SDK se inicializa

  2. SDK guides user to capture document photo
     SDK guia al usuario a capturar foto del documento

  3. SDK captures selfie/video for liveness
     SDK captura selfie/video para liveness

  4. SDK compresses + uploads images to object storage (S3)
     SDK comprime + sube imagenes al object storage

  5. SDK calls: POST /verifications
     {
       customer_id: "bank-prod",
       document_type: "passport",
       country: "DE",
       images: {
         doc_front: "s3://bucket/doc-front.jpg",
         doc_back: "s3://bucket/doc-back.jpg",
         selfie: "s3://bucket/selfie.jpg"
       },
       callback_url: "https://customer.com/webhook"
     }

  6. Platform returns: 202 Accepted + verification_id
     La plataforma responde: 202 Accepted + verification_id

  7. SDK polls or uses SSE for progress updates
     SDK hace polling o usa SSE para actualizaciones

  8. When done: platform POSTs result to callback_url
     Al terminar: la plataforma envia resultado al callback_url

Components explained / Componentes explicados

ComponentResponsibility / ResponsabilidadTopic link
API Gateway Entry point. Auth, rate limiting, creates workflow instances. Returns 202 + ID. Topic 7: Proposals
Workflow Selector Picks the right DAG definition based on country, document type, and customer config. Topic 6: Branching
DAG Engine Your executeWorkflow(). Resolves dependencies, runs steps in parallel, tracks clock ticks. Home (What You Built)
Workflow Context Shared data bag. Steps read inputs and write outputs. Passed by reference, not copy. Topic 1: Data Flow
Decision Engine Evaluates business rules on all step outputs. Returns APPROVED / REJECTED / MANUAL_REVIEW. Topic 3: Decision
PostgreSQL Persists workflow state after each step. Enables crash recovery. Topic 5: Persistence
Object Storage (S3) Stores images, face crops, video frames. Steps pass URIs, not bytes. Topic 1: Data Flow
Event Log Append-only audit trail. Every state change recorded. Legally required (eIDAS, AMLR). Topic 5: Persistence / Topic 8: Observability

Layer 3: External Services / Servicios Externos

These are internal AI/ML and compliance services behind REST endpoints. Your orchestrator calls them — you don't build them. Each follows the standard step contract.

Estos son servicios internos de IA/ML y compliance detras de endpoints REST. Tu orquestador los llama — tu no los construyes. Cada uno sigue el contrato estandar de pasos.

ServiceInput / EntradaOutput / SalidaTime
Document AI (OCR) Document image URI Extracted fields (name, DOB, doc number, expiry), confidence score, face crop URI 2-10s
Liveness + PAD Selfie/video URI alive=bool, confidence, best frame URI, PAD score 1-5s
Face Match Two face image URIs (from OCR + liveness) match_score (0-1) 1-3s
Sanctions + PEP Name, DOB, nationality hit=bool, match details, requires_manual_review flag 1-15s
IAD (Injection Attack Detection) Video stream metadata injected=bool, confidence 1-3s
Video Ident All previous outputs + video call link agent_verdict, notes Minutes (async)
Standard contract / Contrato estandar

Every service follows the SAME interface. The orchestrator sends POST /execute with {workflow_id, step_id, inputs} and gets back {status, outputs, error}. No special code per service.

Cada servicio sigue la MISMA interfaz. El orquestador envia POST /execute y recibe {status, outputs, error}. Sin codigo especial por servicio.

Where data lives — summary / Donde viven los datos

What / QueWhere / DondeWhy / Por que
Document images (front, back) S3 Object Storage Binary files, too large for DB. SDK uploads directly. Services read via URI.
Selfie / video S3 Object Storage Same — binary media. Liveness service reads from S3.
Face crop (extracted by OCR) S3 Object Storage OCR service creates it, saves to S3, returns URI. Face Match reads it.
Best frame (from liveness) S3 Object Storage Liveness picks the best frame, saves to S3. Face Match reads it.
Workflow state (status, step results) PostgreSQL ACID transactions. Updated after each step. Enables crash recovery.
Step outputs (name, DOB, scores) PostgreSQL (JSONB) Structured data, small. Stored as JSONB for flexibility.
Audit events PostgreSQL (Event Log) Append-only. Every state change. Legally required (eIDAS, AMLR).
Final verdict PostgreSQL APPROVED / REJECTED / MANUAL_REVIEW + reason. Queryable for analytics.
The rule / La regla

Binary files (images, video) → S3. Structured data (status, scores, decisions) → PostgreSQL. Steps never pass raw bytes — they pass S3 URIs. This keeps the orchestrator lightweight and the database small.

Archivos binarios (imagenes, video) → S3. Datos estructurados (estado, puntajes, decisiones) → PostgreSQL. Los pasos nunca pasan bytes — pasan URIs de S3. Esto mantiene el orquestador ligero y la base de datos pequena.

When does the DB get written? / Cuando se escribe en la BD?
  1. Workflow created → INSERT workflow_instances + INSERT workflow_events
  2. After each clock tick → UPDATE step_executions + INSERT workflow_events (per step)
  3. Decision made → UPDATE workflow_instances (verdict) + INSERT workflow_events
  4. Each write is a transaction — never a half-written state

1. Workflow creado → INSERT. 2. Despues de cada clock tick → UPDATE + INSERT eventos. 3. Decision tomada → UPDATE + INSERT. 4. Cada escritura es una transaccion.

What to Say on the Whiteboard / Que decir en la pizarra

"Walk me through the system architecture."
"The system has three layers. At the top: clients — mobile SDKs, web SDK, and customer backends. They interact with our REST API. In the middle: our platform — the API gateway handles auth and routing, the workflow orchestrator runs a DAG of verification steps, and the data layer persists state to PostgreSQL with images in object storage. At the bottom: external services — internal AI/ML teams behind REST endpoints. OCR, liveness detection with PAD, face matching, sanctions screening. Every service follows a standard contract: we send workflow_id + step_id + inputs, they return status + outputs. The orchestrator doesn't have service-specific code — adding a new service means defining a new step in the DAG, not changing the engine."
"El sistema tiene tres capas. Arriba: clientes — SDKs moviles, web SDK, backends de clientes. Hablan con nuestra API REST. En el medio: nuestra plataforma — el gateway maneja auth y routing, el orquestador ejecuta un DAG de pasos de verificacion, y la capa de datos persiste estado en PostgreSQL con imagenes en object storage. Abajo: servicios externos — equipos de IA/ML detras de endpoints REST. OCR, liveness con PAD, face matching, sanciones. Cada servicio sigue un contrato estandar. El orquestador no tiene codigo especifico por servicio — agregar un servicio nuevo es definir un paso en el DAG, no cambiar el motor."
"What communication protocol do you use between layers?"
"Everything is REST. Clients call our API via POST/GET. Our orchestrator calls services via POST /execute. Results come back synchronously with a generous timeout per service type. For the client layer, we return 202 Accepted immediately and offer SSE for real-time progress updates plus a webhook callback when verification completes. If a specific service proves too slow, we can migrate that ONE integration to async with a callback — but start simple."
"Todo es REST. Los clientes llaman nuestra API via POST/GET. El orquestador llama servicios via POST /execute. Resultados sincrónicos con timeout generoso. Para el cliente, 202 Accepted inmediato + SSE para progreso en tiempo real + webhook cuando termina. Si un servicio resulta muy lento, migramos ESA integracion a async — pero empezar simple."
"Where does data live?"
"Three stores, all in EU. PostgreSQL for workflow state — instances, step results, decisions. Object storage (S3-compatible) for binary data — document images, face crops, video frames. Steps pass URIs, never raw bytes. And an append-only event log for audit trail — every state change recorded with timestamps. The event log is legally required for eIDAS and AMLR compliance."
"Tres almacenes, todo en la UE. PostgreSQL para estado del workflow. Object storage para datos binarios — imagenes, face crops, video. Los pasos pasan URIs, nunca bytes. Y un log de eventos append-only para auditoria — cada cambio de estado registrado. El log es legalmente requerido por eIDAS y AMLR."

Why a monorepo matters here

The step-contracts package is shared with the AI/ML and compliance teams. When you change the contract, everyone updates together — no version mismatch across services.

El paquete step-contracts se comparte con los equipos de IA/ML y compliance. Cuando cambias el contrato, todos actualizan juntos — sin problemas de versiones.
