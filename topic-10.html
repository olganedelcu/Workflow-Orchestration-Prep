Topic 10: Webhook Event Details

Detalles de Eventos Webhook • High Priority

What this page covers / Que cubre esta pagina

This is the real-world webhook contract — the exact format of events your platform sends to customers when something happens. This is what Revolut, Deutsche Bank, etc. receive at their callback_url.

Este es el contrato real del webhook — el formato exacto de los eventos que tu plataforma manda a los clientes cuando algo pasa. Esto es lo que Revolut, Deutsche Bank, etc. reciben en su callback_url.

Remember / Recuerda

A webhook is just: "when something happens on MY side, I send a POST to YOUR URL". The customer gives us their URL. We call it when events happen. See Architecture for the callback_url explanation.

Un webhook es solo: "cuando algo pasa en MI lado, mando un POST a TU URL". El cliente nos da su URL. Nosotros la llamamos cuando pasan eventos.

How Webhook Delivery Works / Como funciona el envio de webhooks

YOUR PLATFORM                              CUSTOMER'S SERVER
(IDnow)                                    (e.g. Revolut)
    │                                           │
    │  Something happens:                       │
    │  user verified, updated, etc.             │
    │                                           │
    │── POST https://api.revolut.com/webhooks ─▶│
    │   Headers:                                │
    │     Content-Type: application/json        │
    │     x-unizo-event-type: user:created      │  ← what happened
    │     x-unizo-webhook-id: whk_abc123        │  ← which webhook config
    │     x-unizo-delivery-id: del_xyz789       │  ← unique per delivery (idempotency)
    │     x-unizo-signature: sha256=a1b2c3...   │  ← proof it's from us
    │   Body:                                   │

    │                                           │
    │◀── 200 OK ────────────────────────────────│
    │                                           │

The Headers Explained / Los headers explicados

HeaderTypeRequiredWhat it does / Que hace
Content-Type string Yes Always application/json
x-unizo-event-type string Yes What happened: user:created, user:updated, user:deleted
x-unizo-webhook-id string Yes Which webhook configuration this delivery belongs to. The customer might have multiple webhooks configured.
x-unizo-delivery-id string Yes Unique per delivery. The customer uses this for idempotency — if they receive the same delivery-id twice, they know it's a retry and can skip processing it again.
x-unizo-signature string Yes HMAC SHA-256 signature. Proves the webhook really came from us and wasn't tampered with. The customer verifies it using the webhook secret we gave them during onboarding.
Why the signature matters / Por que importa la firma

Without the signature, anyone could send a fake POST to Revolut's webhook URL saying "user verified = APPROVED". The HMAC signature proves it came from YOUR platform. Same concept as the JWT signature — cryptographic proof of authenticity.

Sin la firma, cualquiera podria mandar un POST falso diciendo "usuario verificado". La firma HMAC prueba que vino de TU plataforma. Mismo concepto que la firma JWT — prueba criptografica de autenticidad.

Webhook Delivery & Retries

Reintentos de entrega — que pasa cuando el webhook falla

If the customer's server is down or returns an error, we don't just give up. We retry with increasing delays:

Si el servidor del cliente esta caido o devuelve un error, no nos rendimos. Reintentamos con retrasos crecientes:

RETRY SCHEDULE:
───────────────
  Initial delivery:  Immediate        (right when the event happens)
  1st retry:         After 1 minute
  2nd retry:         After 5 minutes
  3rd retry:         After 15 minutes
  Final retry:       After 1 hour

  Total: 5 attempts over ~1 hour 21 minutes

  CRONOGRAMA DE REINTENTOS:
  Entrega inicial:   Inmediata
  1er reintento:     Despues de 1 minuto
  2do reintento:     Despues de 5 minutos
  3er reintento:     Despues de 15 minutos
  Ultimo reintento:  Despues de 1 hora

A webhook delivery is considered failed if:

Failure conditionWhat it means / Que significa
Non-2xx status codeCustomer's server returned an error (400, 500, etc.)
Connection timeout (30s)Customer's server didn't respond within 30 seconds
SSL/TLS errorsCertificate problems on the customer's side
This is OUR retry (platform → customer), not service retry

Don't confuse this with the retry logic in Topic 4: Failures. That topic covers retrying calls to external services (OCR, liveness). THIS retry is about delivering the final result to the customer's webhook URL. Different problem, same pattern (exponential backoff).

No confundas esto con los reintentos del Topic 4. Ese topic cubre reintentos a servicios externos (OCR, liveness). ESTE reintento es sobre entregar el resultado final al webhook del cliente. Problema diferente, mismo patron (backoff exponencial).

Best Practices for Webhook Consumers

Mejores practicas para los que reciben webhooks — what we recommend to our customers

These are recommendations we give to customers integrating with our webhooks. In an interview, mentioning these shows you think about the consumer side, not just the producer.

Estas son recomendaciones que damos a los clientes. En una entrevista, mencionarlas muestra que piensas en el lado del consumidor, no solo del productor.

1. Idempotency / Idempotencia

The same webhook might be delivered twice (network glitch, retry). The customer must handle duplicates safely using the x-unizo-delivery-id header.

El mismo webhook puede entregarse dos veces (fallo de red, reintento). El cliente debe manejar duplicados usando el header x-unizo-delivery-id.

// Idempotent webhook handler — customer's code
async function handleWebhook(request) {
  const deliveryId = request.headers['x-unizo-delivery-id'];

  // Check if already processed
  // Verificar si ya se proceso
  if (await isProcessed(deliveryId)) {
    return { status: 200, message: 'Already processed' };
  }

  // Process webhook
  await processWebhook(request.body);

  // Mark as processed
  // Marcar como procesado
  await markProcessed(deliveryId);

  return { status: 200 };
}
Why this matters / Por que importa

Imagine we send user:created and the customer's server processes it but crashes before returning 200. We think it failed, so we retry. Without idempotency, the customer creates the user TWICE. With the delivery-id check, the second delivery is safely ignored.

Imagina que mandamos user:created y el servidor del cliente lo procesa pero se cae antes de devolver 200. Nosotros creemos que fallo y reintentamos. Sin idempotencia, el cliente crea el usuario DOS VECES. Con el check de delivery-id, la segunda entrega se ignora.

2. Async Processing / Procesamiento asincrono

The customer should return 200 immediately and process the webhook in the background. Don't do heavy work inside the handler — we might timeout waiting.

El cliente debe devolver 200 inmediatamente y procesar el webhook en segundo plano. No hagas trabajo pesado dentro del handler.

// Asynchronous processing — customer's code
app.post('/webhooks/identity', (req, res) => {
  // Step 1: Validate signature (fast)
  if (!verifySignature(req)) {
    return res.status(401).send('Invalid signature');
  }

  // Step 2: Queue for background processing (fast)
  identityQueue.add(req.body);

  // Step 3: Return immediately (fast)
  // The actual work happens in the queue worker
  res.status(200).send('OK');
});
The same pattern we use / El mismo patron que usamos nosotros

Notice this is the exact same pattern as our POST /verifications → 202 Accepted. Return fast, process in the background. The pattern repeats at every level of the system.

Es el mismo patron que nuestro POST /verifications → 202 Accepted. Devolver rapido, procesar en segundo plano. El patron se repite en cada nivel del sistema.

3. Error Handling / Manejo de errores

Handle unknown event types gracefully. We might add new events in the future — the customer's code shouldn't crash.

Manejar tipos de eventos desconocidos. Podemos agregar nuevos eventos en el futuro — el codigo del cliente no deberia fallar.

// Comprehensive error handling — customer's code
async function processWebhook(payload) {
  try {
    switch (payload.type) {
      case 'user:created':
        await handleUserCreated(payload);
        break;
      case 'user:updated':
        await handleUserUpdated(payload);
        break;
      case 'user:deleted':
        await handleUserDeleted(payload);
        break;
      default:
        // DON'T crash on unknown types — just log it
        // NO fallar con tipos desconocidos — solo registrarlo
        logger.warn(`Unknown webhook type: ${payload.type}`);
    }
  } catch (error) {
    logger.error('Webhook processing failed', {
      error: error.message,
      payload,
      stack: error.stack
    });
    throw error;  // Will trigger our retry
  }
}

"How do you notify the customer when a verification is complete?"
"We use webhooks. When the customer creates a verification, they provide a callback_url. When the workflow completes, we POST the result to that URL with a standard event format — event type, version, payload, and an HMAC signature so they can verify it's really from us. We include a unique delivery ID for idempotency, and we retry with exponential backoff if delivery fails — up to 5 attempts over about an hour. We also recommend customers process webhooks asynchronously: validate the signature, queue the event, return 200 immediately."
"Usamos webhooks. Cuando el cliente crea una verificacion, da un callback_url. Cuando el workflow termina, mandamos un POST con el resultado — tipo de evento, version, payload, y una firma HMAC para que verifiquen que es de nosotros. Incluimos un delivery ID unico para idempotencia, y reintentamos con backoff exponencial si falla — hasta 5 intentos en una hora. Recomendamos a los clientes procesar webhooks de forma asincrona: validar firma, encolar el evento, devolver 200 inmediatamente."
"What if the customer's webhook endpoint is down?"
"We retry with increasing delays: immediately, then 1 minute, 5 minutes, 15 minutes, and finally 1 hour. After all retries are exhausted, we mark the delivery as failed and the customer can query the result via GET /verifications/:id. The verification result is always stored in our database regardless of webhook delivery status — the webhook is a convenience, not the only way to get the result."
"Reintentamos con retrasos crecientes: inmediato, luego 1 min, 5 min, 15 min, y 1 hora. Despues de agotar los reintentos, marcamos la entrega como fallida y el cliente puede consultar el resultado via GET /verifications/:id. El resultado siempre esta en nuestra base de datos — el webhook es una conveniencia, no la unica forma de obtener el resultado."
