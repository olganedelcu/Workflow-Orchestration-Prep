Topic 3: The Decision Step

El Paso de Decision • High Priority

Why It's Special / Por que es especial

Every other step calls an EXTERNAL service. The decision step is YOURS — it's internal business logic that the platform owns. It's the brain of the workflow.

Todos los demas pasos llaman a un servicio EXTERNO. El paso de decision es TUYO — es logica de negocio interna que la plataforma controla. Es el cerebro del workflow.

What It Does / Que hace

INPUTS (from previous steps):
  ocr.confidence:        0.97
  ocr.expiry:            "2028-05-15"
  liveness.alive:        true
  liveness.confidence:   0.95
  face_match.score:      0.92
  sanctions.hit:         false

          |
          v

    ┌─────────────┐
    │  DECISION    │
    │  ENGINE      │
    │  Apply rules │
    └──────┬──────┘
           |
           v

OUTPUT:
  verdict: "APPROVED"
  reason:  "All checks passed with high confidence"

The Rules / Las reglas

Think of rules as a checklist evaluated top to bottom. First match wins.

Piensa en las reglas como una lista evaluada de arriba a abajo. La primera que coincida gana.

RULE 1 (highest priority):
  IF sanctions.hit == true
  THEN → REJECTED  "Sanctions match found"

RULE 2:
  IF liveness.alive == false
  THEN → REJECTED  "Liveness check failed"

RULE 3:
  IF face_match.score < 0.5
  THEN → REJECTED  "Face does not match document"

RULE 4:
  IF face_match.score < 0.8
  THEN → MANUAL_REVIEW  "Low confidence face match"

RULE 5:
  IF ocr.confidence < 0.7
  THEN → MANUAL_REVIEW  "Document hard to read"

RULE 6:
  IF ocr.expiry < today
  THEN → REJECTED  "Document expired"

DEFAULT:
  → APPROVED  "All checks passed"

Hardcoded vs Configurable

Option A: Hardcoded rules in code

function decide(outputs: AllStepOutputs): Decision {
  if (outputs.sanctions.hit) return { verdict: "REJECTED", reason: "..." };
  if (!outputs.liveness.alive) return { verdict: "REJECTED", reason: "..." };
  if (outputs.faceMatch.score < 0.5) return { verdict: "REJECTED", reason: "..." };
  return { verdict: "APPROVED", reason: "All checks passed" };
}
ProsCons
Simple, easy to testNeed code deploy to change rules
Type-safeProduct team can't change rules themselves

Option B: Configurable rules (decision table)

{
  "rules": [
    { "condition": "sanctions.hit == true",    "verdict": "REJECTED",      "priority": 1 },
    { "condition": "liveness.alive == false",  "verdict": "REJECTED",      "priority": 2 },
    { "condition": "face_match.score < 0.5",   "verdict": "REJECTED",      "priority": 3 },
    { "condition": "face_match.score < 0.8",   "verdict": "MANUAL_REVIEW", "priority": 4 },
    { "condition": "ocr.confidence < 0.7",     "verdict": "MANUAL_REVIEW", "priority": 5 }
  ],
  "default_verdict": "APPROVED"
}
ProsCons
Change rules without deployingMore complex to implement
Product team can adjust thresholdsNeed to validate rule syntax
Different rules per country/customerHarder to test all combinations
What to propose

"Start with hardcoded rules for v1 — simpler, testable, ship faster. But design the interface so we can swap in a configurable rules engine later. The orchestrator just calls decide(allOutputs) and doesn't care how the decision is made internally."

"Empezar con reglas fijas para v1 — mas simple, testeable, entregar rapido. Pero disenar la interfaz para que podamos cambiar a un motor de reglas configurable despues."

Important: Not Every Bad Result Is a Failure

Key insight / Punto clave
OCR returns confidence: 0.4    → NOT a step failure
                                  It's a low score. Step COMPLETED.
                                  The DECISION step handles it.

OCR service returns HTTP 500   → THIS is a step failure
                                  Retry or fail the workflow.

Separate infrastructure failures from bad results. The orchestrator handles infrastructure failures. The decision step handles bad results.

Separa fallos de infraestructura de malos resultados. El orquestador maneja fallos de infraestructura. El paso de decision maneja malos resultados.

"How do you handle the business logic for the decision step?"
"I'd separate orchestration from decision logic. The orchestrator feeds all previous step outputs as inputs. The decision step evaluates rules in priority order. For v1, hardcoded. The interface is clean enough to swap in a configurable rules engine later — important if different customers or countries need different thresholds. Key insight: a low confidence score isn't a system failure — the step still completes successfully. It's the decision step's job to interpret what that score means."
"Separaria la orquestacion de la logica de decision. Para v1, reglas fijas. La interfaz permite cambiar a un motor configurable despues. El punto clave: un puntaje bajo no es un fallo del sistema — el paso completa exitosamente. Es trabajo del paso de decision interpretar que significa."
